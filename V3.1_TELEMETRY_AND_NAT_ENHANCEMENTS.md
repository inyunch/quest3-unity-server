## What's New in V3.1

### 1. Server Timestamp Propagation Fix

**Problem**: `queue_wait_ms` always showed 0 in CSV telemetry

**Root Cause**:
- Server calculated `queue_wait_ms` correctly in response
- Unity was **recalculating** from timestamps instead of using server value
- Timestamps were all 0, causing recalculation to fail

**Solution** (3 files modified):

**Server Side** (`vision_server`):
```python
# app/core/inference/base.py - Add server timestamp fields
@dataclass
class InferenceResult:
    server_receive_ts: int = 0           # When server received frame
    server_process_start_ts: int = 0     # When inference started
    server_send_ts: int = 0              # When server sent response

# app/workers/udp_inference_worker_v3.py - Set timestamps
inference_result.server_receive_ts = int(req.server_receive_ts)
inference_result.server_process_start_ts = int(server_process_start_ts)
result['server_send_ts'] = int(time.time() * 1000)  # Just before UDP send
```

**Unity Side**:
```csharp
// FrameTrace.cs - Add field
public float queue_wait_ms;  // Time waiting in server admission queue

// FrameTelemetryTracker.cs - Store server value directly
trace.queue_wait_ms = response.queue_wait_ms;  // Don't recalculate!

// LocalTelemetryWriter.cs - Use trace value
trace.queue_wait_ms.ToString("F2"),  // No more calculation
```

**Impact**:
- ✅ queue_wait_ms now shows accurate values (e.g., 0-5ms in V3.0 vs 101ms in V2.0)
- ✅ Enables proper analysis of server queue admission performance
- ✅ All 3 modes (Detection, Pose, Segmentation) fixed

### 2. Late Response Telemetry for Dropped Frames

**Problem**: Dropped frames had `server_receive_ts = 0` in CSV

**Scenario**:
1. Unity sends frame N via UDP
2. Frame N+5 arrives first, gets displayed
3. Frame N is marked as "Dropped" (superseded)
4. Unity writes frame N to CSV with `server_receive_ts = 0`
5. Frame N response arrives 200ms later with full server timestamps
6. Too late - CSV already written!

**Solution**:

```csharp
// FrameTrace.cs - Don't overwrite Dropped state
public void MarkCompleted(long receiveTime)
{
    // CRITICAL: If already dropped, keep that state (don't overwrite to Completed)
    if (state == FrameState.Dropped)
        return;  // Response arrived late, but frame was already superseded

    state = FrameState.Completed;
    unity_receive_ts = receiveTime;
    e2e_ms = (unity_receive_ts - unity_send_ts) / 1000f;
}

// FrameTelemetryTracker.cs - Detect and rewrite late responses
public void MarkFrameCompleted(int frameId, FrameResponse response)
{
    // Remember original state
    FrameState originalState = trace.state;

    // Update timing (won't change Dropped state due to FrameTrace.cs fix above)
    trace.MarkCompleted(receiveTime);
    trace.server_receive_ts = response.server_receive_ts;
    trace.server_proc_ms = response.processing_time_ms;
    trace.queue_wait_ms = response.queue_wait_ms;

    // If this is a late response for already-dropped frame, rewrite CSV
    if (originalState == FrameState.Dropped)
    {
        Debug.Log($"[TELEMETRY] Late response for dropped frame {frameId}, " +
                  $"server_proc_ms={trace.server_proc_ms:F1}ms, rewriting CSV");
        WriteLocalTelemetry(trace);  // Overwrite previous row
    }
}
```

**Impact**:
- ✅ Dropped frames now have complete server timing data
- ✅ Can distinguish "dropped but processed" vs "network lost"
- ✅ Better understanding of server latency vs Unity display latency

### 3. NAT Hole Punching for 5G Hotspot

**Problem**: UDP responses blocked in 5G hotspot environment

**Scenario**:
- Unity sends frame from random port (e.g., 54321) to Server:8002 ✅ Works
- Unity listens on port 8003 for responses
- NAT only maps port 54321 (outbound), NOT port 8003
- Server response to 8003 is blocked as "unsolicited inbound traffic" ❌

**Solution**: Heartbeat-based NAT hole punching

**Architecture**:
```
Quest 3 (5G Hotspot)          NAT Device          Server (Cloud)
     |                             |                      |
     | Bind UDP socket to 8003     |                      |
     |--------------------------->|                      |
     |                             |                      |
     | Send HEARTBEAT to 8002      |                      |
     |------------------------->| Map 8003↔8002    |
     |                             |---------------->|    |
     | (Heartbeat every 2 sec)     | Keep mapping alive   |
     |<-------------------------|<----------------|    |
     | Receive response on 8003 ✅                      |
```

**Implementation**:

```csharp
// UDPTransportManager.cs - Add heartbeat thread
private Thread m_heartbeatThread;

public void Initialize()
{
    // ... existing code ...

    // Start background heartbeat (NAT hole punching)
    m_heartbeatThread = new Thread(HeartbeatLoop)
    {
        IsBackground = true,
        Name = "UDP NAT Heartbeat"
    };
    m_heartbeatThread.Start();
}

private void HeartbeatLoop()
{
    byte[] heartbeat = Encoding.UTF8.GetBytes("HEARTBEAT");  // 9 bytes
    IPEndPoint serverEndpoint = new IPEndPoint(IPAddress.Parse(m_serverIP), m_sendPort);

    while (m_isRunning)
    {
        try
        {
            // CRITICAL: Use receive socket to send (creates NAT mapping)
            m_receiveClient.Send(heartbeat, heartbeat.Length, serverEndpoint);
            Thread.Sleep(2000);  // Send every 2 seconds
        }
        catch (Exception e)
        {
            if (m_isRunning)
                Debug.LogWarning($"[UDP HEARTBEAT] Send failed: {e.Message}");
            Thread.Sleep(2000);
        }
    }
}
```

**Server Side** (No changes required):
- Server ignores heartbeat packets (< 70 bytes minimum frame size)
- Normal frame processing continues unchanged

**Impact**:
- ✅ Works in 5G hotspot environments (NAT/firewall)
- ✅ Zero impact on WiFi environments (harmless heartbeat)
- ✅ No server-side changes required
- ✅ NAT mapping kept alive indefinitely (2-second refresh)

## Updated Architecture

### V3.1 UDP Flow with NAT Hole Punching

```
Unity (Quest 3)                Server (Cloud)
     |                              |
     | 1. Initialize UDP Transport  |
     |    - Bind send socket        |
     |    - Bind receive socket:8003|
     |    - Start heartbeat thread  |
     |                              |
     | 2. Heartbeat Loop (2sec)     |
     |    HEARTBEAT (9 bytes)       |
     |---------------------------->| (Ignored, < 70 bytes)
     |                              |
     | 3. Frame Cadence (100ms)     |
     |    JPEG + metadata          |
     |---------------------------->|
     |                              | 4. Admission Queue
     |                              | 5. UDP Worker pulls frame
     |                              | 6. Run inference
     |                              |    - Set server_receive_ts
     |                              |    - Set server_process_start_ts
     |                              | 7. Store result in cache
     |                              |    - Set server_send_ts
     |                              |    - Calculate queue_wait_ms
     | 8. Poll for result (HTTP)    |
     |<----------------------------|
     | FrameResponse (JSON)         |
     | - server_receive_ts ✅      |
     | - server_process_start_ts ✅|
     | - server_send_ts ✅         |
     | - queue_wait_ms ✅          |
     |                              |
     | 9. Update telemetry          |
     |    - Store server values     |
     |    - Write to CSV            |
     |    - If dropped, rewrite CSV |
```
